03 Learning objectives

  • Provision Application Insights with Bicep + azd in a single command
  • Auto-instrument an ASP.NET Core 10 Web app with zero code
  • Correlate browser pageView → server request → SQL dependency on one operation_Id
  • Author a KQL query that joins requests, dependencies, and pageViews end-to-end
  • Tear down all resources cleanly with azd down --purge --force

Anchor expectations. Every objective is reinforced by a hands-on lab.

06 Module 1 — Provision (azd up)

  • azd init -t . (use the workshop template)
  • azd env new mapaq-dev
  • azd up (Bicep deploys RG, AI, Log Analytics, App Service, SQL)
  • Check outputs: APPLICATIONINSIGHTS_CONNECTION_STRING is wired to both apps

Expect ~6 min for the first run. The connection string is set as an App Setting, not in code.

07 Bicep modules in this workshop

  • infra/main.bicep — composes the modules
  • infra/modules/log-analytics.bicep — workspace foundation
  • infra/modules/app-insights.bicep — workspace-based component
  • infra/modules/app-service.bicep — Web + Api with system-assigned managed identity
  • infra/modules/sql.bicep — Azure SQL with Entra-only auth
  • Pass connection string via App Setting, never hard-code

Reinforce that AI is now workspace-based by default; classic AI is deprecated.

08 Module 2 — Web auto-instrumentation

  • dotnet add package Microsoft.ApplicationInsights.AspNetCore
  • builder.Services.AddApplicationInsightsTelemetry();
  • Reads APPLICATIONINSIGHTS_CONNECTION_STRING from configuration automatically
  • Generates requests, dependencies, exceptions, and traces with zero extra code

This is intentionally boring. The magic is the next module — propagation.

09 What you should see in 60 seconds

  • Live Metrics shows incoming requests in real time
  • Transaction Search lists every request with its operation_Id
  • End-to-end transaction view groups dependencies under their parent request
  • Failures blade lights up with any 5xx the API returns

Keep Live Metrics open during the demo; nothing convinces a room faster than dots appearing.

10 Module 3 — API + SQL instrumentation

  • Same Microsoft.ApplicationInsights.AspNetCore package on the API
  • EF Core 10 SQL calls are auto-tracked as SqlClient dependencies
  • HttpClientFactory propagates traceparent to downstream calls automatically
  • Custom dimensions: TelemetryClient.TrackEvent("OrderPlaced", new { OrderId, CustomerId })

EF Core ⇒ SqlClient instrumentation requires no extra package — it ships with the AI SDK.

11 W3C Trace Context — traceparent

  • 00-{trace-id}-{parent-id}-{trace-flags}
  • Browser SDK starts the trace and emits the header on every fetch/XHR
  • ASP.NET Core honours and forwards it without configuration
  • Result: one operation_Id stitches all 4 spans together

Show the actual header in the browser DevTools network tab during the demo.

13 Module 4 — Browser correlation

  • Add the Application Insights JavaScript SDK snippet to _Layout.cshtml
  • Pass the connection string via a server-side ViewData entry (never hard-code)
  • Enable enableCorsCorrelation: true and enableRequestHeaderTracking: true
  • Pass distributedTracingMode: AI_AND_W3C to emit traceparent

The snippet must come before any other script that issues fetch/XHR.

14 Browser SDK — minimal config

  • instrumentationKey is gone — use connectionString
  • enableAutoRouteTracking: true (for SPA-style page changes)
  • correlationHeaderExcludedDomains: ['*.googleapis.com', '*.bing.com']
  • samplingPercentage: 100 in dev, lower in prod

Sampling is per-app, set on the SDK options object — not in the portal.

16 KQL — one query, one operation_Id

  • Open Logs blade on the AI resource
  • Run the union query below
  • Sort by timestamp asc to see the waterfall in time order
flowchart LR
  Q["union requests, dependencies, pageViews

| order by timestamp asc"]
  Q --> R["📊 Waterfall result

Walk through each line. The union across 3 tables is the trick most newcomers miss.

17 KQL query — copy / paste

  • union requests, dependencies, pageViews
  • | where operation_Id == 'abc-trace-id-here'
  • | project timestamp, itemType, name, duration, success, target
  • | order by timestamp asc

Replace 'abc-trace-id-here' with the operation_Id from the browser DevTools traceparent header.

20 Smart alerts — failure anomalies

  • Smart Detection runs continuously on the workspace
  • Auto-detects failure-rate anomalies, slow page loads, and trace-degradation
  • Configure email + Action Group for Teams webhook
  • Rule of thumb: alert on symptoms (user impact), not causes

Smart Detection is on by default — show how to subscribe additional emails.

21 Cost controls — no surprises

  • Daily cap on the AI resource (50 MB/day for the workshop)
  • Sampling: 100% in dev, 10–25% in prod typical
  • Workspace-based AI uses Log Analytics commitment tiers
  • Tag everything with Workshop=Mapaq for billing slicing

The daily cap protects the workshop subscription from runaway log volume.

22 Troubleshooting checklist

  • No telemetry? Verify APPLICATIONINSIGHTS_CONNECTION_STRING in App Settings
  • Browser pageViews missing? Confirm the snippet is before other scripts
  • operation_Id mismatch? Check enableCorsCorrelation on the JS SDK
  • SQL deps missing? EF Core needs the AI SDK — package alone is not enough

Keep this slide visible during the lab; most questions map to one of these four lines.

23 Module 6 — Teardown

  • azd down --purge --force
  • --purge removes soft-deleted Key Vaults and AI resources
  • --force skips the interactive confirmation prompt
  • Verify the resource group is gone in the Azure portal

Always demo teardown live. A workshop without teardown is a billing time-bomb.
